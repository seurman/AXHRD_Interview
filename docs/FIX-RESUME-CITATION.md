# 버그 수정: 자소서 인적사항이 면접 질문 인용문으로 잘못 뽑히던 문제

- **발견일**: 2026-07-05
- **영향 범위**: 자소서 맞춤 질문 생성 전체 (`personalizeQuestion`) — Gemini 개인화 경로와, API 키 없을 때 쓰는 휴리스틱 경로 모두

## 문제 상황

금융 업계 · HR 직무로 면접을 시작했을 때 실제로 나온 첫 질문:

> 금융 업계 HR 직무 관점에서 자소서에 적어 주신 「또한 매주 진행한 스터디 모임에서
> 분산 시스템 설계 사례를 발표하고 동료들과 토론하며 이론을 실제 아키텍처 결정에
> 적용하는 연습을 했습니다.」 경험과 「성명 김가람 지원 부문 SW 개발 / 신입 학력
> 고려대학교 컴퓨터공학과 졸업 (2026.2) 연락처 example@email.com 1.」 성과를
> 바탕으로 여쭤봅니다. 복잡한 내용을 비전문가에게 설명했던 경험이 있나요?

앞의 「」 인용(스터디 모임 경험)은 정상이지만, 뒤의 「」 인용은 실제 "성과"가
아니라 **자소서 맨 위 인적사항(성명·지원부문·학력·연락처·이메일) 한 줄을
그대로 인용**한 것 — 명백히 맥락을 잘못 짚은 결과였다.

## 원인

`web/src/lib/interview/personalize-question.ts`의 `extractResumeHighlights()`가
자소서 문장 중 "성과"로 인용할 후보를 아래 정규식으로 골랐다:

```ts
const withMetrics = lines.filter((s) => /\d+%?|\d+명|\d+억|\d+건/.test(s));
```

`\d+%?`는 `%`가 **선택(optional)** 이라 사실상 숫자만 있으면 전부 매칭되는
허술한 패턴이었다. 그리고 이력서 상단 인적사항 블록은 보통 마침표로 문장이
끊기지 않고 한 줄로 붙어 있는데(`성명 OOO 지원 부문 OOO 학력 OOO 졸업 (2026.2)
연락처 OOO@OOO.com`), 이 문장 분리 로직이 이 블록을 걸러내는 필터가 없었다.

결과: "졸업 (2026.2)"의 "2026.2"나 이메일 속 숫자가 "성과 지표"로 오인되어,
`heuristicPersonalize()`가 이걸 그대로 질문 문구에 박아 넣었다.

## 수정 내용

`web/src/lib/interview/personalize-question.ts` 한 파일 수정, 두 가지:

1. **인적사항 블록 필터 추가** — 이메일·전화번호 패턴이 있거나, "성명/이름·지원
   부문/분야·학력·연락처·생년월일·전화번호·이메일" 같은 이력서 헤더 라벨이
   **한 줄에 2개 이상** 나오면 애초에 인용 후보(`splitResumeSentences`)에서
   제외한다 (`CONTACT_INFO`, `HEADER_LABEL`, `isHeaderMetaLine()`).

2. **"성과" 판단 기준 강화** — 단순히 숫자가 있다고 성과로 보지 않고, `%· 퍼센트·
   명·건·억·만원·천만원·배·시간·일·개월·주(년)·회` 같은 실제 단위가 붙은
   경우만 성과로 인정하도록 `METRIC_PATTERN`을 새로 정의하고,
   `extractResumeHighlights()`와 `heuristicPersonalize()` 양쪽 모두 이 기준으로
   통일했다 (전에는 `heuristicPersonalize()`가 `/\d/.test(h)`로 더 허술하게 따로
   판단하고 있었음).

## 확인 방법

이력서 맨 위에 "성명/지원부문/학력/연락처/이메일" 같은 인적사항이 있는 자소서로
면접을 다시 시작해서, 첫 질문에 그 인적사항 줄이 더 이상 인용되지 않는지 확인.
(자소서 본문에 실제 STAR형 성과 문장 — 예: "매출 20% 개선", "3개월 만에 완료" —
이 있다면 그 문장이 대신 인용되는 게 정상.)
